# v0.7.9 Implementation Plan

**Status:** Drafted 2026-05-23. Implementation to land in a separate
session per author cadence (per-step commits, alignment between
sub-tasks).

**Scope per [ADR-0019 §A7.9][adr-7.9]:**

1. `compiler/pack_writer.tri` — port of `crates/triet-pack/src/serde.rs`
   (1303 LOC) + `crates/triet-ir/src/serde.rs` (3249 LOC); emit
   `.triv` + `.tripack` byte streams.
2. `compiler/main.tri` — CLI driver wires the full
   `lex → parse → modules → typecheck → lower → pack` pipeline
   entirely in Triết, in-memory (no NDJSON bridge).
3. **Drop NDJSON bridges** introduced in v0.7.4–v0.7.8: the parser /
   modules / typecheck / lowerer differential tests stop calling
   their `dump_*_ndjson` helpers from the in-memory pipeline path
   (the bridge functions stay as test-only utilities until
   v0.7.9.5 removes them).
4. End-to-end test: Triết-impl reads a `.tri` source, emits a
   `.tripack`, byte-compares against the Rust-impl-emitted
   `.tripack`. This is the precondition for v0.7.11 Stage 2
   bootstrap.

## Pre-flight bugs investigated 2026-05-23

The two pinned "ValueKind kind-tag loss" gotchas from v0.7.8.4 and
v0.7.8.6 turned out **not** to be VM-side bugs. Root-cause
investigation (15+ minimal Rust unit tests in
`crates/triet-ir/tests/` — now deleted as no longer needed) showed
the lowerer's ValueKind propagation through `bind_var` /
`rebind_var` / chained field access works correctly. The actual
fault was logic-level:

### Fixed: `lower_module` dropped `ctx.constants` on the floor

**Symptom:** `dump_ir_program_ndjson("function f() -> Integer = 42")`
emitted `"const":"Unit"` instead of `"const":"Integer(42)"`. The
v0.7.8.5 byte-equality smoke kept failing.

**Root cause:** `lower_module` (in `compiler/ir_lowerer.tri`)
returned `IrModule` only — the `LowerCtx.constants` pool
accumulated across all function lowerings was discarded.
`lower_source` built the final `IrProgram` via `ir_program_new()`
(empty pool) + `ir_program_push_module(m)` without ever copying
the threaded `ctx.constants` over. Every `Const` instruction's
`ConstId` reference dangled — `lookup_constant(prog, c0)` walked
into an empty `prog.constants.entries` vector and returned null,
which `dump_const_value` mapped to the literal `"Unit"` string.

**Fix (2-line change in `compiler/ir_lowerer.tri`):**

1. Added `LowerModuleResult { ir_module: IrModule, constants:
   ConstantPool }` struct + made `lower_module` return it instead
   of bare `IrModule`.
2. Updated `lower_source` to lift the result's `constants` onto
   the final `IrProgram` via the existing
   `ir_program_with_constants` setter.

**Verification:** v0.7.8.5 byte-equality smoke restored (tests
1, 2, 3, 4 in `ir_dump_smoke_main` now assert exact NDJSON
including `"const":"Integer(42)"`). 1393 workspace tests pass,
clippy clean.

**Lesson learned:** the "ValueKind kind-tag loss" theory was based
on observed-but-unverified symptoms across two sub-tasks. Future
investigations should attempt isolated Rust unit-test repros
**first** before assuming a VM-level bug — saves 1-2 sessions of
defensive helper-pattern accumulation.

### Still pinned: `v0.7.x.runtime-fix.propagate-capture` (blocking-soft)

**Symptom:** `expr ~? |capture| early_return` form where `expr` is
a non-trivial Outcome path produces a runtime
`TypeMismatch { expected: HashMap(Unit, Unit), actual: "Unit",
function: "declare_with_mut" }`. Observed during v0.7.7.4b smoke
test for `infer_outcome_propagate`.

**Status:** Not investigated as part of the 2026-05-23 fix
session. Likely also a logic-level bug (e.g. the typecheck-side
capture binding registers the wrong kind in `Frame.names`), not
VM-level. Repro plan: isolated Rust unit test mimicking the
`function f(o: T~E) -> T~E = o ~? |e| ~- e` pattern, assert no
runtime panic during VM execution.

**Impact on v0.7.9:** the serde port has many error paths shaped
as `T ~? |e| ~- LowerError::SomeKind(e)`. If unresolved, v0.7.9
would need to rewrite every error-propagating call site to manual
match-and-rethrow (~500 LOC boilerplate). Recommend
investigating before v0.7.9.1 opens.

### Pinned: `v0.7.x.runtime-fix.const-items` (not blocking v0.7.9)

Pre-existing in `TODO.md`. `Item::Const` lowering missing. v0.7.9
shouldn't need top-level constants either, deferral OK.

### Pinned: `v0.7.x.runtime-fix.cli-std-resolution` (blocking-soft)

**Symptom observed during v0.7.9 planning:**

```
$ ./target/release/triet run /path/to/test.tri
triet::runtime::E2002

  × undefined name `concat`
```

Yet `concat` is imported via `from std.text import concat`. The
`triet check` command on the same file returns OK. This means the
*runtime* path resolves stdlib imports differently from the
*check* path.

**Impact on v0.7.9:** `compiler/main.tri` IS the new CLI driver
written in Triết — it will hit the same path. If `triet run`
can't resolve stdlib imports consistently, the self-hosted CLI
can't run either.

**Investigation plan:**
1. Compare `triet-cli` `run` subcommand vs `check` subcommand for
   stdlib search-path resolution.
2. Decision: either land stdlib pre-load *now* as
   v0.7.x.runtime-fix.stdlib-preload (pulls v0.7.10 forward), or
   document that `compiler/main.tri` is invoked from inside a
   project directory with explicit `std/` alongside.

## Sub-task breakdown

5 sub-tasks following the v0.7.5 / .6 / .7 / .8 cadence.

### v0.7.9.1 — `.triv` writer + reader scaffolding

**Files:**
- `compiler/pack_writer.tri` (new, ~500-700 LOC)
- `crates/triet-bootstrap/tests/triv_serde_smoke.rs` (new)

**Scope:**
1. LEB128 / varint primitives (`write_u32_leb`, `read_u32_leb`,
   etc.) matching `crates/triet-ir/src/serde.rs` byte layout.
2. `write_constant_pool` / `read_constant_pool` round-trip for
   each `Constant` variant (8 variants).
3. `write_type_tag` / `read_type_tag` recursive over `TypeTag`
   arena (11 variants — primitives + Nullable/Vector/HashMap/Outcome).
4. **Defer**: full Instruction encoder — too big for one
   sub-task, lands in v0.7.9.2.

**Acceptance:** `triv_serde_smoke.rs` round-trips a synthetic
ConstantPool + TypeTag arena Triết→bytes→Rust→bytes→Triết and
asserts byte-identical at each hop.

**Estimated LOC:** ~600 Triết + ~120 Rust test.

### v0.7.9.2 — Instruction + Function + Module encoder/decoder

**Files:**
- `compiler/pack_writer.tri` (+1500 LOC)
- `crates/triet-bootstrap/tests/triv_round_trip.rs` (new)

**Scope:**
1. Per-instruction byte encoder for all 53 Instruction variants.
   Opcode byte (0x00-0x3F per ADR-0008 §opcode table) + payload
   layout. Mirror `crates/triet-ir/src/serde.rs:1112-1500`.
2. PhiIncoming list encoding.
3. AbsolutePath encoding (path segments + name).
4. Block encoding: terminator-last invariant + length prefix.
5. Function encoding: parameters + return type + blocks.
6. IrModule encoding: path + functions list.
7. IrProgram encoding: modules + constants + witness tables.
8. Full decoder mirror — reverse of every encoder.

**Acceptance:** `triv_round_trip.rs` lowers a corpus of 10+
sources (covering every Instruction variant), Triết-writes the IR
to bytes, Rust-reads the bytes back, asserts identity with the
original Rust-lowered IR.

**Estimated LOC:** ~1500 Triết + ~150 Rust test.

### v0.7.9.3 — `.tripack` assembly (ABI metadata + sections)

**Files:**
- `compiler/pack_writer.tri` (+600 LOC)
- `crates/triet-bootstrap/tests/tripack_serde_smoke.rs` (new)

**Scope:**
1. ABI metadata header per `triet-pack/src/serde.rs:88-120` —
   magic bytes + version + iface_hash + impl_hash placeholders.
2. Code section assembly: concat `.triv` payload from v0.7.9.2.
3. Caps section emission (reserved per ADR-0011, populated per
   ADR-0016 §4 — `caps section` carrying `CapabilityLevel` per
   declared namespace from `triet.package`).
4. Sort-by-name canonical encoding per ADR-0019 §3.
5. BLAKE3 hash computation — likely needs a Rust-side builtin
   exposed via `call_builtin` since BLAKE3 in pure Triết is too
   slow at v0.7. **Decision needed**:
   - Option A: add builtin opcode `blake3_hash(bytes) -> bytes`
     to Rust VM (extends ADR-0019 §5 builtin table).
   - Option B: leave hash placeholder, compute at link step from
     content (v0.7.10 scope).
   - **Recommendation:** Option A — small VM extension, unlocks
     CAS write path for Stage 2 bootstrap.

**Acceptance:** `tripack_serde_smoke.rs` writes a `.tripack` for a
single-function program, reads it back via
`triet_pack::read_tripack`, asserts the ABI metadata + code
section match.

**Estimated LOC:** ~600 Triết + ~150 Rust test + ~30 Rust VM
builtin if Option A.

### v0.7.9.4 — `compiler/main.tri` CLI driver

**Files:**
- `compiler/main.tri` (new, ~400-500 LOC)
- `crates/triet-bootstrap/tests/triet_in_triet_compile.rs` (new)

**Scope:**
1. `parse_args(argv: Vector<String>) -> Args` — subcommand
   dispatch (`check` / `build` / `run` / `dump`).
2. `check_command(args: Args) -> ExitCode` — reads source, runs
   `lex → parse → modules → typecheck`, emits diagnostics to
   stderr.
3. `build_command(args: Args) -> ExitCode` — adds
   `lower → pack_write` to the pipeline, writes `.tripack` to
   output path.
4. `run_command(args: Args) -> ExitCode` — `build` + execute via
   VM (needs `vm_execute` builtin or defer to v0.7.11 Stage 2).
5. `dump_command(args: Args) -> ExitCode` — emit lowered IR for
   debugging (uses v0.7.8.5 NDJSON dump pre-deletion; this is the
   last consumer of the bridge).

**Decision needed before opening:** `run_command` requires a VM
hook callable from Triết code. Options:
- **A.** Add `call_builtin vm_execute(bytes, fn_name) -> exit_code`
  to Rust VM. Self-hosted Triết compiler can execute its own
  output. Recommended.
- **B.** Defer `run_command` — Triết compiler only emits
  `.tripack`; user invokes a separate `triet-runner` (Rust-side
  binary) to actually run it.

**Acceptance:** `triet_in_triet_compile.rs` invokes the
self-hosted compiler's `main()` on the same `examples/factorial.tri`
source the Rust compiler processes, asserts the two `.tripack`
outputs byte-match.

**Estimated LOC:** ~500 Triết + ~200 Rust test + ~50 Rust VM
builtin if Option A.

### v0.7.9.5 — Drop NDJSON bridges + end-to-end gate

**Files:**
- All `compiler/*.tri` — strip `dump_*_ndjson` functions
  (transitional per ADR-0019 §A2).
- `crates/triet-bootstrap/tests/*_differential.rs` — replace the
  cached IR + NDJSON-VM-dump pattern with direct in-memory calls
  to the now-public lex/parse/check/lower API.
- `crates/triet-bootstrap/tests/bootstrap_self_compile.rs` (new) —
  the actual gate.

**Scope:**
1. Confirm each differential test still passes after replacing
   NDJSON dump with structural comparison via direct Triết-side
   function calls. The differential test pattern stays
   (opcode-coverage gates from v0.7.8.6) but the *transport* is
   in-memory.
2. Delete `dump_program_ndjson` (parser), `dump_resolved_program_ndjson`
   (modules), `dump_typecheck_errors_ndjson` (typecheck),
   `dump_ir_program_ndjson` (ir_lowerer), and all the
   per-instruction `dump_inst_*` helpers from
   `compiler/ir_lowerer.tri`.
3. New gate: `bootstrap_self_compile.rs` invokes `compiler/main.tri`
   on `compiler/main.tri` itself (Stage 2 input), asserts the
   output `.tripack` is byte-identical to the Rust-lowered version
   of `compiler/main.tri`. **This is the v0.7.9 acceptance gate.**

**Acceptance:** `cargo test --workspace` green with NDJSON
bridges removed. `bootstrap_self_compile.rs` passes.

**Estimated LOC:** -2000 Triết (deletions) + +150 Rust test.

## Design questions to resolve before v0.7.9.1

Per ADR-0019 cadence — author decides before opening sub-task.

1. **`.triv` serde port faithfulness**: 1:1 port of Rust impl
   (3000+ Triết LOC, faithful) vs. simplified subset covering only
   opcodes shipped in v0.7.8 (~1500 LOC, gap documented). The
   simplified subset can grow as new opcodes ship in v0.8+.
   - **Recommendation:** start with 1:1 port. Self-hosted compiler
     must read/write `.tripack` files the Rust toolchain produces,
     and v0.8 may not be backward-compatible with a subset.

2. **End-to-end byte-identical gate scope**:
   - **A.** `compiler/factorial.tri` (trivial, single fn).
     v0.7.9.5 acceptance.
   - **B.** Subset of `compiler/*.tri` (5 of 7 files). Tighter
     gate.
   - **C.** Full self-compile of all 7 compiler files. This is
     Stage 2 from §1 — defer to v0.7.11.
   - **Recommendation:** A for v0.7.9.5 acceptance, C for v0.7.11
     gate. B can be a v0.7.10 intermediate if needed.

3. **`vm_execute` builtin in Rust VM**: add or defer?
   - **A.** Add. Enables `run_command` from Triết, simpler test
     story.
   - **B.** Defer. v0.7.9.4 only ships `check_command` +
     `build_command`; running uses external Rust runner.
   - **Recommendation:** B — minimal scope for v0.7.9, add the
     builtin in v0.7.11 when Stage 2 needs it.

4. **`blake3_hash` builtin**: same A vs B as above.
   - **Recommendation:** A — small VM extension (~30 LOC), unlocks
     the CAS write path which `triet build` needs. Without it,
     `compiler/main.tri::build_command` can't compute
     `iface_hash` / `impl_hash` and falls back to placeholders,
     breaking the byte-identical gate.

5. **`propagate-capture` runtime bug**: investigate + fix as
   prerequisite, or work around with manual match-and-rethrow?
   - **Recommendation:** investigate first (estimated 2-4 hours
     based on the const-pool-drop fix complexity). Likely a
     logic-level bug in `lower_outcome_propagate` similar in
     spirit to the const-pool drop — not a VM bug.

## Appendix: v0.7.8.4 accessor-helper pattern (now obsolete)

During v0.7.8.4 and v0.7.8.6 implementation, the const-pool drop
bug was misdiagnosed as a VM ValueKind propagation issue. The
pinned workaround introduced typed accessor helpers
(`ctx_current_block_idx(ctx: LowerCtx) -> Integer = ctx.current_block_idx`,
`ctx_blocks(ctx: LowerCtx) -> Vector<BasicBlock> = ctx.blocks`,
`ctx_loop_stack(ctx: LowerCtx) -> Vector<LoopContext> = ctx.loop_stack`).

These helpers are **functionally inert** — they do nothing the
direct field access wouldn't do. They remain in
`compiler/ir_lowerer.tri` as v0.7.x.cleanup candidates; the
v0.7.9 session should:

1. Replace all `ctx_current_block_idx(st)` calls with
   `st.current_block_idx` direct access.
2. Same for `ctx_blocks` and `ctx_loop_stack`.
3. Delete the 3 helper definitions.
4. Verify all 1393 workspace tests still pass.

Net LOC delta: ~−50.
